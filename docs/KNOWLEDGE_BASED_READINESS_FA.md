## معرفی محصول

Padyar Live Avatar یک پلتفرم آواتار دیجیتال زنده است که امکان تعامل بصری و صوتی بلادرنگ با هوش مصنوعی را فراهم می‌کند. این سیستم از یک موتور تولید تصویر (lip-sync) استفاده می‌کند تا حرکات لب آواتار را بر اساس ورودی صوتی یا متنی به‌صورت آنی تولید کند.

## هدف محصول

ایجاد یک زیرساخت بلادرنگ و قابل‌استقرار برای آواتارهای دیجیتال هوشمند که بتواند در محیط‌های سازمانی، آموزشی، پشتیبانی مشتری و تعامل انسانی-ماشینی مورد استفاده قرار گیرد.

## ارزش فناورانه

- معماری چندلایه با جداسازی کامل مسئولیت‌ها
- runtime بدون وابستگی به فریمورک‌های یادگیری ماشین
- رابط `EngineAdapter` برای اتصال هر موتور استنتاج خارجی
- تست‌های خودکار حاکمیتی (governance tests) برای تضمین سلامت معماری
- پشتیبانی از WebSocket برای استریم بلادرنگ تصاویر
- مدیریت نشست (session lifecycle) با TTL، تشخیص بیکاری و پاکسازی خودکار
- ردیابی تأخیر (latency) با محاسبه avg، p95 و p99
- بازیابی خودکار هنگام خطا یا قطع موتور استنتاج (fallback recovery)

## نوآوری‌ها

1. **runtime بدون ML:** لایه اجرایی هیچ وابستگی مستقیم به torch، tensorflow، onnxruntime یا هر فریمورک یادگیری ماشین ندارد. تمام دسترسی به موتور استنتاج از طریق رابط `EngineAdapter` انجام می‌شود.

2. **معماری جداکننده (Separation of Concerns):** هر لایه مسئولیت مستقلی دارد و هیچ وابستگی دایره‌ای بین آن‌ها وجود ندارد. این معماری توسط تست‌های خودکار تأیید می‌شود.

3. **حاکمیت خودکار (Automated Governance):** مجموعه‌ای از تست‌ها وجود دارند که هرگونه نقض معماری (وارد کردن فریمورک ML، وابستگی دایره‌ای، دسترسی مستقیم به موتور) را شناسایی و مسدود می‌کنند.

4. **بازیابی خودکار (Fallback Recovery):** در صورت قطع یا خطای موتور استنتاج، runtime به‌صورت خودکار فریم جایگزین تولید می‌کند و خدمات قطع نمی‌شود.

5. **SDK موبایل بومی:** Android SDK با پیاده‌سازی بومی C++/NCNN برای lip-sync آفلاین روی دستگاه، و iOS SDK با رندرینگ Metal.

## تفاوت بخش‌های مختلف مخزن

| بخش | نقش | وضعیت |
|---|---|---|
| `src/padyar_live/` | runtime بلادرنگ — FastAPI، WebSocket، مدیریت نشست، زمان‌بند فریم، تأخیر، fallback | فعال، تحت حاکمیت |
| `tests/` | تست‌های runtime — 54 تست شامل حاکمیت، قرارداد، REST، WebSocket، زمان‌بند، نشست، تأخیر | فعال، همه پاس |
| `mobile/padyar-android/` | SDK اندروید — NCNN بومی، lip-sync آفلاین، استریم PCM | مرجع، build متوقف |

## چرایی ML-free بودن runtime

runtime عمدتاً هیچ وابستگی به فریمورک‌های یادگیری ماشین ندارد. دلایل:

1. **استقرار مستقل:** runtime بتواند بدون GPU اجرا شود
2. **تست بدون سخت‌افزار:** با `FakeEngineAdapter` امکان تست کامل بدون کارت گرافیک وجود دارد
3. **مقیاس‌پذیری مستقل:** runtime و موتور استنتاج به‌صورت جداگانه مقیاس‌پذیر هستند
4. **تیم مستقل:** هر لایه توسط تیم جداگانه توسعه و نگهداری می‌شود

## نقش EngineAdapter

رابط `EngineAdapter` تنها نقطه ارتباط بین runtime و موتور استنتاج است:

- `generate_frames()` — تولید فریم‌های تصویری
- `health_check()` — بررسی سلامت موتور

هر موتوری که این رابط را پیاده‌سازی کند، بدون تغییر در runtime قابل اتصال است.

## وضعیت تست‌ها و حاکمیت

- **ruff:** بدون خطا
- **mypy:** بدون خطا در 18 فایل منبع
- **pytest:** 54 تست، همه پاس

تست‌های حاکمیت شامل:
- عدم وجود فریمورک ML در کد و وابستگی‌ها
- عدم دسترسی مستقیم به موتور استنتاج
- عدم وابستگی دایره‌ای بین لایه‌ها
- رعایت قرارداد `EngineAdapter`
- عدم وارد کردن از `mobile/` یا `PadYar-LipSync-master/`

## مسیر دانش‌بنیان

این محصول دارای ویژگی‌های زیر برای احراز صلاحیت دانش‌بنیان است:

1. **نوآوری فناورانه:** معماری runtime بدون ML با رابط جایگزینی موتور
2. **دانش فنی تخصصی:** پیاده‌سازی lip-sync بومی روی موبایل با NCNN
3. **مالکیت فکری:** معماری و طراحی اختصاصی
4. **بلوغ فناوری:** تست‌های خودکار و حاکمیت معماری
5. **قابلیت تجاری‌سازی:** کاربردهای متعدد صنعتی
