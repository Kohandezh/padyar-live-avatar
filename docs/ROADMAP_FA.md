## وضعیت پایه فعلی

- مخزن در حالت پایدار روی شاخه `main` قرار دارد
- runtime بدون وابستگی ML، تحت حاکمیت کامل، با 54 تست پاس‌شده
- ادغام monorepo کامل شده و در main ادغام شده است
- SDK موبایل اندروید و iOS به‌عنوان مرجع در مخزن موجود هستند
- اسنپشات تاریخی موتور استنتاج به‌عنوان مرجع حفظ شده است
- مستندات دانش‌بنیان تهیه شده است

## Build موبایل — متوقف

- build اندروید و iOS از ابتدا متوقف شده و بدون تأیید صریح نباید از سر گرفته شود
- شناسه‌های فنی قدیمی (guiji/gj/GJ/GJL) در SDK موبایل به‌عنوان شناسه‌های build-critical طبقه‌بندی شده‌اند
- هیچ مرجع مستقیم Duix در SDK موبایل یافت نشده

## طرح آینده تغییر نام موبایل (Mobile Rebrand)

- **وضعیت:** نیاز به طرح تأیید‌شده جداگانه
- **محدوده:**
  - تغییر namespace جاوا/Kotlin در Android (`ai.guiji.padyar`)
  - تغییر نام کتابخانه بومی Android (`libgjpadyar.so`)
  - تغییر JNI bindings
  - تغییر نام targets و classes در iOS (`GJLocalDigital*`)
  - تغییر bundle identifiers در iOS (`com.wanruyi.*`)
  - بازسازی `GJLDecry.framework` از منبع
  - به‌روزرسانی URLهای دانلود مدل
- **ریسک:** تغییر نادرست باعث شکست build می‌شود. نیاز به تست کامل روی هر پلتفرم
- **پیش‌نیاز:** تأیید صریح مالک محصول

## طرح آینده ادغام EngineAdapter

- **وضعیت:** نیاز به تأیید صریح
- **محدوده:**
  - تقویت قرارداد `EngineAdapter`: lifecycle اتصال، طبقه‌بندی خطا، نسخه‌بندی
  - ساخت HTTP/WebSocket bridge به موتور PadYar-LipSync
  - تست یکپارچگی با موتور واقعی
  - پیکربندی endpoint موتور
- **ریسک:** وابستگی به در دسترس بودن موتور استنتاج
- **پیش‌نیاز:** موتور PadYar-LipSync در دسترس و مستقر شده باشد

## طرح آینده استقرار و Docker

- **وضعیت:** پیش‌بینی شده، نیاز به تأیید
- **محدوده:**
  - Dockerfile برای runtime
  - docker-compose برای توسعه محلی
  - CI/CD pipeline
  - پیکربندی محیطی (staging, production)
  - مانیتورینگ و logging
- **ریسک:** وابسته به زیرساخت هدف
- **پیش‌نیاز:** تأیید محیط استقرار

## لایه محصول آینده — PadYarAvatar

- **وضعیت:** شروع نشده، خارج از محدوده این مخزن
- **محدوده پیش‌بینی‌شده:**
  - حافظه مکالمه‌ای
  - شخصیت و عاطفه آواتار
  - رفتار عامل هوشمند (agent behavior)
  - یکپارچگی با LLM
- **ریسک:** وابسته به بلوغ runtime و موتور استنتاج
- **پیش‌نیاز:** runtime و موتور در سطح تولید باشند

## ریسک‌ها و کنترل‌ها

| ریسک | کنترل |
|---|---|
| وارد کردن تصادفی ML در runtime | تست‌های خودکار `test_governance_imports` و `test_governance_deps` |
| وابستگی دایره‌ای بین لایه‌ها | تست `test_governance_architecture` |
| دسترسی مستقیم به موتور بدون adapter | قرارداد `EngineAdapter` و تست `test_adapter_contract` |
| شکست build موبایل پس از تغییر نام | طرح مجزا با تست روی هر پلتفرم قبل از merge |
| قطع خدمات در production | بازیابی خودکار (fallback) در runtime |
| ادامه build موبایل بدون تأیید | قوانین صریح در CLAUDE.md و مستندات |

## درب‌های تأیید (Approval Gates)

هر مرحله پیاده‌سازی نیاز به تأیید صریح مالک محصول دارد:

1. **قبل از شروع هر phase جدید:** تأیید مکتوب مورد نیاز است
2. **قبل از commit:** بررسی تغییرات
3. **قبل از push:** تأیید نهایی
4. **قبل از merge:** تأیید صریح
5. **قبل از از سرگیری build موبایل:** تأیید صریح جداگانه
6. **قبل از تغییر نام شناسه‌های فنی:** طرح تأیید‌شده مجزا
